The story

I’m not a programmer. The Rust agent harness running on AMD MI300X for our submission was built without me opening an IDE or writing a single line of Rust. Eight days of work, all from a Claude.ai chat tab on a Windows machine. Here’s how the workflow actually runs.

Three layers

Layer 1 — chat. I describe what I want. The chat (currently Claude Opus 4.7) thinks alongside me, asks clarifying questions, names the architectural decisions, and condenses what I said into focused implementation prompts.

Layer 2 — Manager MCP. A server I run locally that lets the chat delegate work to other AI agents. The chat calls manager:session_start with a task description, working directory, and effort level. Manager spawns a Claude Code process in the background, hands it the task, returns a session ID. The chat keeps responding to me while the code agent grinds. A Windows toast fires when it’s done.

Layer 3 — Claude Code, on my machine or via SSH on the AMD droplet. Reads files, writes files, runs cargo build, edits configs, restarts tmux sessions, smoke-tests endpoints, commits to git. Reports back with files changed and test results when the work is done.

The chat is the architect. The code agent is the contractor. I’m the homeowner.

A typical hour

Yesterday afternoon I wanted closed-loop verify+retry on the agent: when an operation finishes with a stated success signal, a critic model audits the result. If the result doesn’t satisfy the signal, the system dispatches one corrective retry with an intelligent hint and escalates if a second attempt fails. Two models, four new HTTP endpoints, around 200 lines of Rust across five files.

I described requirements in chat. The chat condensed them into a 60-line prompt with constraints, archive-first instructions, and smoke tests. Fired manager:session_start with effort=high. I made coffee. Claude Code edited agent.rs, critic.rs, serve.rs, config.rs, run.rs. Cargo-built. Restarted the driver tmux session. Smoke-tested all four endpoints. Reported back 30 minutes later.

Then we tested live: prompt “what is 2+2, spell it as a word” with success_signal: "reply contains the digit 4". The agent replied “Four.” Critic verdict came back fail — reasoning “contains the word ‘Four’ but lacks the digit ‘4’”, corrective hint “instruct the model to output the digit instead of spelling out the word.” A retry dispatched automatically. The retry came back “4”. Verification pass.

That JSONL trace — actual events from a real run — is in the AIWander/AMDHack repo as proof. The architecture I described in chat became running code that catches semantic mismatches and corrects itself. Without me touching the keyboard except to type messages.

Surviving compactions

Chat hits its memory limit and compacts. Half the context disappears. This would be catastrophic if I didn’t have a layer beneath it: a CPC ecosystem with three patterns that mattered.

Volumes — Google-Drive-synced markdown files organized by topic. Project state lives in projectstate/aiwander.md. Detailed handoffs go in handoffs/2026-05-10/<timestamp>_<topic>.md. Append-only, NAV-indexed. After a compaction, the next message reads these and reconstructs.

Project Loaf — structured JSON tracking phases, decisions, and discoveries. Chat updates it as work progresses. Cross-session recovery anchor.

Breadcrumbs — multi-step operations in flight. If chat crashes mid-build, the next session picks up at step N+1.

These aren’t novel — they’re CPC patterns I built up over months. The point is they existed before AIWander, and that’s what made AIWander viable from a chat window.

When delegation works and when it doesn’t

Manager-delegated Claude Code is excellent for focused implementation tasks (5-500 lines), 20-60 minute sustained-focus work units, repetitive operations like build-test-restart loops, and anything I can describe precisely.

It’s bad at ambiguous architectural decisions, picking between equally valid approaches, and knowing when to stop. If the prompt isn’t tight, the session wanders. A few times this week, sessions came back saying “I noticed your prompt was unclear about X — made assumption Y, here’s what I built.” Sometimes Y was right. Sometimes I had to redo the work. Signal-to-noise is high but not perfect.

Why the MI300X mattered

192 GB of HBM3 on a single chip. That’s not a benchmark number — it’s the difference between actor and critic on one GPU and actor here, critic there, network hop between them.

Both reasoning models run co-resident: gpt-oss-120B as the actor (70 GB weights + 15 GB KV cache) plus Qwen3.6-35B-A3B at q8_0 with 256K context as the critic (36 GB weights + 2 GB KV cache). Plus driver and Next.js overhead. Roughly 111 GB used, 81 GB free. NVIDIA’s biggest single consumer GPU tops at 80 GB. The two-model architecture wouldn’t fit.

Vendors like to quote compute throughput. I had a workflow that needed memory in one address space. The MI300X disappeared as a constraint and let me build the architecture I actually wanted.
